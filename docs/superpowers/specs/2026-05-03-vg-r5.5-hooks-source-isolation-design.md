## 1. Background

### 1.1 Problem

VG harness installs 7 hook scripts via `.claude/settings.json`. Several hooks fail to gate cleanly when the active Claude Code session is **not running a VG command**, producing two failure modes:

1. **False blocks on non-VG flows** — `vg-pre-tool-use-agent.sh` rejects subagent spawns whose `subagent_type` is outside the VG allow-list, even when the user is running an unrelated skill (e.g. `superpowers:dispatching-parallel-agents`, `gsd:execute-phase`). The hook does not check whether a VG run is active before enforcing the allow-list.
2. **Hard session failure** — `vg-session-start.sh` calls `exit 1` if `vg-meta-skill.md` is missing. This kills the SessionStart hook chain for the entire Claude Code session, not only VG flows.

The `.vg/active-runs/${session_id}.json` file is the canonical source of truth for "is a VG run currently active". Hooks that already check it (`vg-pre-tool-use-bash.sh`, `vg-stop.sh`, `vg-post-tool-use-todowrite.sh`, `vg-user-prompt-submit.sh`) silently exit when the file is absent. The 3 broken hooks must adopt the same pattern.

### 1.2 Scope of fix

**In scope** (audit + patch):
- `scripts/hooks/vg-pre-tool-use-agent.sh` — add VG context guard before allow-list enforcement
- `scripts/hooks/vg-session-start.sh` — graceful degrade when meta-skill missing
- `scripts/hooks/vg-pre-tool-use-write.sh` — **explicit decision** required: keep enforcement (defensive) or gate behind VG context

**Out of scope** (already correct, regression test only):
- `vg-pre-tool-use-bash.sh` — has guard at lines 12-21
- `vg-stop.sh` — has guard at lines 9-12
- `vg-user-prompt-submit.sh` — has guard at line 14 (only matches `/vg:` prompts)
- `vg-post-tool-use-todowrite.sh` — has guard at lines 10-12

### 1.3 Audit findings (verified 2026-05-03)

| Hook | Active-run guard | Decision |
|---|---|---|
| `vg-pre-tool-use-bash.sh` | exit 0 if no run_file (line 18-21) | **OK — regression test only** |
| `vg-pre-tool-use-write.sh` | NONE — runs protected-path check on every Write/Edit | **DECIDE: defensive vs gated** (see §3.3) |
| `vg-pre-tool-use-agent.sh` | NONE — runs allow-list on every Agent spawn | **PATCH — add guard** |
| `vg-session-start.sh` | meta-skill missing → exit 1 | **PATCH — graceful degrade** |
| `vg-user-prompt-submit.sh` | exit 0 if prompt does not match `/vg:` (line 14) | **OK — regression test only** |
| `vg-post-tool-use-todowrite.sh` | exit 0 if no run_file (line 10-12) | **OK — regression test only** |
| `vg-stop.sh` | exit 0 if no run_file (line 9-12) | **OK — regression test only** |

### 1.4 Goals

- All 7 hooks are silent (`exit 0`, no stderr) when invoked outside a VG run, with one explicit exception (`vg-pre-tool-use-write.sh` defensive mode — see §3.3).
- `vg-session-start.sh` never aborts a Claude Code session, even when `vg-meta-skill.md` is absent or unreadable.
- Pytest regression suite locks the silent-on-non-VG behavior so future hook edits cannot regress.
- One-line entry in `scripts/hooks/vg-meta-skill.md` documents the "VG context guard" pattern as required for any new hook.

### 1.5 Non-goals

- Restructuring hook architecture (no new hooks, no removed hooks).
- Changing telemetry events or block-handling semantics inside VG runs.
- Migrating hook implementations from bash to python.
- Modifying `.claude/settings.json` matchers.
- Codex mirror — Codex hooks are separate; defer to dedicated round.

## 3. Patch design

### 3.1 The "VG context guard" pattern

Every hook that enforces VG-specific policy MUST start with this guard block, identical bytes across all hooks for grep-ability:

```bash
# ── VG context guard ──
# Hook is harmless when no VG run is active. Silent exit prevents
# false blocks on unrelated Claude Code skills (superpowers, gsd, etc).
session_id="${CLAUDE_HOOK_SESSION_ID:-default}"
run_file=".vg/active-runs/${session_id}.json"
if [ ! -f "$run_file" ]; then
  exit 0
fi
```

Placement: immediately after `set -euo pipefail` and `input="$(cat)"` (if the hook reads stdin), before any policy check.

### 3.2 `vg-pre-tool-use-agent.sh` patch

**Current behavior** (lines 8-17):
```bash
input="$(cat)"
subagent="$(... extract subagent_type ...)"

if [[ "$subagent" =~ ^(general-purpose|Explore|Plan|gsd-debugger)$ ]]; then
  exit 0
fi
if [[ "$subagent" == vg-* ]]; then
  exit 0
fi
# else: emit_block (allow-list enforcement)
```

**Patched behavior** — insert guard between `input="$(cat)"` and the first allow-list check:
```bash
input="$(cat)"

# ── VG context guard ──
session_id="${CLAUDE_HOOK_SESSION_ID:-default}"
run_file=".vg/active-runs/${session_id}.json"
if [ ! -f "$run_file" ]; then
  exit 0
fi

subagent="$(... extract subagent_type ...)"
# allow-list enforcement only when VG run is active
```

**Rationale**: subagent allow-list is a VG-specific policy (R1a Codex bypass #3 fix). Outside a VG run, every Claude Code skill should be free to spawn whatever subagent it wants.

### 3.3 `vg-pre-tool-use-write.sh` decision

**Current behavior**: blocks ALL writes to protected paths (`.vg/runs/*/evidence-*`, `.vg/events.db`, etc.) regardless of session context.

**Two valid designs**:

**Option A — Keep defensive (recommend)**: write protection is a property of the **filesystem path**, not the session. If any skill (VG or not) writes to `.vg/events.db` outside the signed evidence pipeline, integrity is corrupted. Defensive enforcement protects the data store from accidental writes by any caller. Cost: a non-VG skill that legitimately wants to inspect `.vg/runs/*` for read-only debugging cannot accidentally write — which is correct behavior.

**Option B — Gate behind VG context**: silently allow non-VG flows to write protected paths. Cost: corrupts evidence integrity guarantee.

**Decision**: **Option A**. Keep `vg-pre-tool-use-write.sh` as-is. Add a comment at the top explaining why this hook intentionally lacks the VG context guard:

```bash
# NOTE: this hook does NOT use the VG context guard. Protected-path
# enforcement is filesystem-scoped, not session-scoped. Any caller
# (VG or not) writing to .vg/runs/*/evidence-* or .vg/events.db
# corrupts the signed evidence pipeline. See R5.5 design §3.3.
```

### 3.4 `vg-session-start.sh` graceful degrade patch

**Current behavior** (lines 14-17):
```bash
if [ ! -f "$META_SKILL_PATH" ]; then
  echo "ERROR: meta-skill missing at $META_SKILL_PATH" >&2
  exit 1
fi
base_text="$(cat "$META_SKILL_PATH")"
```

`exit 1` from a SessionStart hook prints a red "Hook failed" line in Claude Code chat and prevents context injection. Innocent users without VG installed see an error every session.

**Patched behavior**:
```bash
if [ ! -f "$META_SKILL_PATH" ]; then
  # Graceful degrade — VG meta-skill missing, no context to inject.
  # Log once per session for diagnostics; do NOT block session.
  warn_log=".vg/.session-start-warn.log"
  mkdir -p "$(dirname "$warn_log")" 2>/dev/null || true
  printf '%s session=%s meta-skill missing at %s\n' \
    "$(date -u +%Y-%m-%dT%H:%M:%SZ)" \
    "${CLAUDE_HOOK_SESSION_ID:-default}" \
    "$META_SKILL_PATH" >> "$warn_log" 2>/dev/null || true
  exit 0
fi
base_text="$(cat "$META_SKILL_PATH")"
```

Same treatment for any future `exit 1` paths in `vg-session-start.sh` (e.g. unreadable meta-skill, malformed JSON in events.db). Failures inside the hook MUST NOT block the session.

### 3.5 Documentation update

Append a new section to `scripts/hooks/vg-meta-skill.md`:

```markdown
## Hook authoring rule — VG context guard

Every NEW hook script that enforces VG-specific policy MUST start with the
canonical "VG context guard" block (see R5.5 design §3.1). Hooks that
enforce filesystem-scoped invariants (write protection on `.vg/runs/*`)
MAY skip the guard, but MUST document the rationale at the top of the
script.

The guard pattern is identical bytes across all hooks for grep-ability:

    session_id="${CLAUDE_HOOK_SESSION_ID:-default}"
    run_file=".vg/active-runs/${session_id}.json"
    if [ ! -f "$run_file" ]; then
      exit 0
    fi

Pytest suite `tests/hooks/test_hooks_silent_on_non_vg.py` enforces this
for all hooks except the documented exceptions.
```

## 6. Error handling, migration, testing

### 6.1 Error handling

The patches REDUCE error surface. There are no new error paths introduced. The graceful-degrade path in `vg-session-start.sh` writes a warning log to `.vg/.session-start-warn.log` for post-hoc diagnostics; failure to write the log is itself swallowed (`|| true`).

### 6.2 Migration

- All hook patches are backward-compatible. Existing VG runs continue to gate identically.
- Sessions that previously hit the `exit 1` in `vg-session-start.sh` (because VG was uninstalled or meta-skill moved) will now succeed silently. This is the intended behavior change.
- No data migration. No artifact format change.

### 6.3 Testing

**Pytest suite** (`tests/hooks/`):

`test_hooks_silent_on_non_vg.py`:
- Fixture: temp directory with NO `.vg/active-runs/*.json`.
- For each hook in `[bash, write, agent, post-todowrite, stop, user-prompt-submit, session-start]`, invoke the hook script with synthetic stdin matching its matcher.
- Assert: exit code == 0, stderr empty.
- Exception: `vg-pre-tool-use-write.sh` is invoked with a non-protected path; assert silent exit. Then invoked with a protected path; assert block (exit 2) regardless of session state.

`test_session_start_graceful_degrade.py`:
- Fixture: temp directory with no `vg-meta-skill.md`.
- Invoke `vg-session-start.sh` with `CLAUDE_HOOK_EVENT=startup`.
- Assert: exit code == 0, stderr does NOT contain "ERROR", `.vg/.session-start-warn.log` exists with one entry.

`test_write_protection_unconditional.py`:
- Fixture: temp directory with NO active-run file.
- Invoke `vg-pre-tool-use-write.sh` with `tool_input.file_path = ".vg/events.db"`.
- Assert: exit code == 2 (blocked), even though no VG run is active.

**Manual reproduce** (in `docs/superpowers/plans/`):
1. Open a fresh Claude Code session in a non-VG project (no `.vg/` directory).
2. Run any skill that spawns a non-allow-list subagent (e.g. `superpowers:dispatching-parallel-agents` with subagent_type=`Plan`).
3. Verify: spawn succeeds, no "PreToolUse-Agent-allowlist" block in stderr.
4. Repeat with `superpowers:brainstorming` invoking subagents — verify silent.

### 6.4 Exit criteria

R5.5 PASSES when ALL of the following hold:

1. `tests/hooks/test_hooks_silent_on_non_vg.py` — all parameterized cases pass.
2. `tests/hooks/test_session_start_graceful_degrade.py` — passes.
3. `tests/hooks/test_write_protection_unconditional.py` — passes (regression for §3.3 decision).
4. `scripts/hooks/vg-meta-skill.md` contains the new "Hook authoring rule" section.
5. Manual reproduce in non-VG project: no false blocks observed across 3 different non-VG skills.
6. Existing VG runs (R1a-R4 dogfood projects) re-execute without behavioral change — gates fire identically before and after patch.

R5.5 is a **prerequisite hotfix** for R6a/R6b. Reasons:

- R6a (`/vg:deploy` redesign) and R6b (`/vg:amend` + `/vg:debug` redesign) both spawn new subagents (`vg-deploy-executor`, `vg-amend-impact-analyzer`, `vg-debug-classifier`).
