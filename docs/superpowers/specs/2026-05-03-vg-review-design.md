## 1. Background

### 1.1 Why review is the heaviest

`commands/vg/review.md` is **7,413 lines** with **39 steps** — the largest single command in VG by both metrics. Two structural pressures:

1. **Multi-modal discovery surface**: 4 profile paths (web-fullstack, web-frontend-only, web-backend-only, mobile-*), each with own discovery + lens dispatch
2. **Lens architecture**: 18 production lens probes (filter, paging, sort, search, URL-state, visual, error-message, CRUD roundtrip, state coherence, auth/session, security surfaces, business-flow…) — must run inside ONE runtime pass per profile (not separate browser passes per lens)

User concern (verbatim from brainstorm):
> "lens injection ép vào không, lens nào cũng phải chạy không, AI có cherry-pick lens không"

### 1.2 Heavy steps (>300 lines)

| Step | Lines | Role | Refactor approach |
|---|---|---|---|
| `phase2_browser_discovery` | **947** | Spawn Haiku scanners ≤5 parallel for view crawl | Dedicated subagent: `vg-review-browser-discoverer` |
| `phase4_goal_comparison` | **829** | Goal → RUNTIME-MAP lookup → binary READY/BLOCKED verdict (NOT weighted scoring per audit) | **Inline ref + split** (no subagent; logic is binary lookup, complexity is from branching not formula) |
| `complete` | 542 | Artifact finalization | Inline ref (split) |
| `phase3_fix_loop` | 414 | Error remediation iteration | Inline ref |
| `phase1_code_scan` | 389 | Static scan + element inventory | Inline ref |
| `0_parse_and_validate` | 317 | Frontmatter audit | Inline ref |
| `phaseP_delta` | 314 | Change-only mode | Inline ref |

### 1.3 Lens dispatch architecture (already exists)

Review **already has a sophisticated lens system**. Recon evidence:

- **18 lens definitions** in `commands/vg/_shared/lens-prompts/` (lens-authz-negative, lens-csrf, lens-idor, lens-input-injection, lens-bfla, lens-business-logic, lens-business-coherence, lens-auth-jwt, lens-duplicate-submit, lens-file-upload, lens-form-lifecycle, lens-mass-assignment, lens-modal-state, lens-open-redirect, lens-path-traversal, lens-ssrf, lens-table-interaction, lens-tenant-boundary)
- **`REVIEW-LENS-PLAN.json` contract** generated by `review-lens-plan.py` from API-CONTRACTS.md + goal bodies. Telemetry: `review.lens_plan_generated` (must_emit)
- **6-precondition eligibility gate** at step `phase2_5_recursive_lens_probe`: profile match, surface type, CRUD declaration, resource intersection, ENV-CONTRACT presence, seed data validity. Skip leaves audit trail in `.recursive-probe-skipped.yaml`
- **3-axis interactive preflight** (MANDATORY_GATE): `RECURSION_MODE` (light/deep/exhaustive), `PROBE_MODE` (auto/manual/hybrid/skip), `TARGET_ENV` (local/sandbox/staging/prod). Anti-forge guard refuses launch if all empty
- **Manager dispatch** via `spawn_recursive_probe.py` (1,303 LOC): element-class → lens mapping (LENS_MAP: 11 classes × 1-3 lenses each)
- **Parallel auto-mode**: Gemini Flash subprocess workers per (element × lens × role) tuple
- **Aggregator** `aggregate_recursive_goals.py`: per-tool subdir isolation, single-writer dedup

**Implication:** Review's lens injection is ALREADY enforced at architecture level. AI cannot cherry-pick lens because `LENS_MAP` is hardcoded in `spawn_recursive_probe.py` and eligibility gate is automatic.

### 1.4 What's NOT enforced (the gap)

- **AI may skip the entire lens phase** by skipping step `phase2_5_recursive_lens_probe` (no per-step PreToolUse gate yet)
- **AI may abuse `--skip-discovery` flag** — currently allowed without override-debt for some step combinations
- **Lens plan generation may be staled** — if API-CONTRACTS.md changes after lens plan generated, AI uses stale plan
- **No telemetry per individual lens** — only `review.lens_plan_generated` and `review.recursive_probe.preflight_asked` exist; no `review.lens.<name>.dispatched` events to verify each of 18 lenses ran

### 1.6 Goals

- Reduce `commands/vg/review.md` from 7,413 → ≤500 lines (Anthropic ceiling)
- Apply imperative + HARD-GATE + Red Flags (review-specific)
- **1 dedicated subagent** for `phase2_browser_discovery` (947 lines, true heavy lifting). `phase4_goal_comparison` (829 lines) is binary lookup → inline ref split (no subagent — verified by audit, no weighted formula exists)
- **Add per-lens telemetry**: emit `review.lens.<name>.dispatched` for each lens dispatched
- **Strengthen eligibility gate observability**: emit `review.lens_phase.entered` and `review.lens_phase.completed` so Stop hook can detect skip
- **Add lens-plan staleness check**: compare REVIEW-LENS-PLAN.json mtime vs API-CONTRACTS.md mtime; block if stale
- Empirically prove on PrintwayV3 phase 3.2 (the existing dogfood UI bug at billing/topup-queue)
- Defer: lens-as-plugin pluggable architecture (nice-to-have, current LENS_MAP works)

### 1.7 Non-goals

- Refactor of `spawn_recursive_probe.py` (1,303 LOC works as-is; only add telemetry hooks)

## 5. Components

### 5.1 Strengthen lens telemetry (audit FAIL items 7, 8, 10)

**Item 7 — Per-lens telemetry:**
In `spawn_recursive_probe.py`, after each lens dispatch (per element × lens × role), emit:
```
vg-orchestrator emit-event review.lens.<lens_name>.dispatched \
  --phase ${PHASE_NUMBER} \
  --element <element_id> \
  --role <role_name> \
  --provider <gemini|codex|haiku>
```
And after aggregation:
```
vg-orchestrator emit-event review.lens.<lens_name>.completed \
  --phase ${PHASE_NUMBER} \
  --findings_count <N> \
  --duration_ms <T>
```

**Item 8 — Lens-phase enter/exit:**
In `phase2_5_recursive_lens_probe` step bash:
```
# At step entry (after eligibility check passes):
vg-orchestrator emit-event review.lens_phase.entered --phase ${PHASE_NUMBER}

# At step end:
vg-orchestrator emit-event review.lens_phase.completed \
  --phase ${PHASE_NUMBER} \
  --lens_count_dispatched <N> \
  --lens_count_completed <M>
```

**Add to runtime_contract.must_emit_telemetry:**
```yaml
- event_type: "review.lens_phase.entered"
  phase: "${PHASE_NUMBER}"
  required_unless_flag: "--skip-discovery"
- event_type: "review.lens_phase.completed"
  phase: "${PHASE_NUMBER}"
  required_unless_flag: "--skip-discovery"
```

→ Stop hook detects: lens phase entered but never completed = unhandled abort. Or: never entered + no `--skip-discovery` flag = AI skipped lens phase.

**Item 10 — Lens-plan staleness:**
At step `phase2_5_recursive_lens_probe` start:
```bash
plan_mtime=$(stat -f %m "${PHASE_DIR}/REVIEW-LENS-PLAN.json")
contracts_mtime=$(stat -f %m "${PHASE_DIR}/API-CONTRACTS.md")
if [ "$contracts_mtime" -gt "$plan_mtime" ]; then
  echo "BLOCK: REVIEW-LENS-PLAN.json is older than API-CONTRACTS.md. Regenerate before continuing." >&2
  exit 2
fi
```

## Red Flags (review-specific)

| Thought | Reality |
|---|---|
| "Lens dispatch tốn thời gian, làm 3-4 cái đại diện đủ"      | LENS_MAP hardcoded, AI không thể cherry-pick. Skip = skip whole phase = block |
| "Browser discovery scan 1 view đại diện đủ"                  | phase2 spawn ≤5 Haiku scanners parallel; per-view evidence required |
| "Goal comparison làm nhanh, skip RUNTIME-MAP lookup" | phase4 binary verdict lookup mandatory — skip = wrong verdict (no formula, but lookup must run) |
| "REVIEW-LENS-PLAN.json đã có rồi, dùng lại"                  | Staleness check vs API-CONTRACTS.md mtime — stale = regenerate |
| "Fix loop iterate 1-2 lần là đủ"                             | exploration limits configurable; legitimate stop only via gate |

## Steps

### STEP 1 — preflight
Read `_shared/review/preflight.md`. Follow exactly.

### STEP 2 — code scan + API precheck
Read `_shared/review/code-scan.md`. API-CONTRACTS.md and api-docs PRECHECK
must run BEFORE discovery (build pipeline contract).

### STEP 3 — browser discovery (HEAVY, subagent)
Read `_shared/review/discovery/overview.md`. Spawn `vg-review-browser-discoverer`
subagent. DO NOT crawl inline.

### STEP 4 — lens dispatch (architectural enforcement)
Read `_shared/review/lens-dispatch.md`. Run eligibility gate, 3-axis preflight,
then invoke `spawn_recursive_probe.py`. Per-lens telemetry emitted automatically.
DO NOT skip individual lenses (LENS_MAP enforces) and DO NOT skip the phase
without override-debt.

### STEP 5 — findings collect + fix loop
Read `_shared/review/findings/collect.md` then `fix-loop.md`. Iterate per
exploration limit config.

### STEP 6 — goal comparison + verdict (inline ref split, NO subagent)
Read `_shared/review/verdict/overview.md`. Branch on profile/UI_GOAL_COUNT
to load matching sub-ref (pure-backend-fastpath / web-fullstack /
profile-branches). Logic is binary lookup against RUNTIME-MAP (READY/BLOCKED) —
no weighted formula exists per audit, so inline split suffices.

### STEP 7 — CrossAI review (UNCHANGED, defer refactor)
Read `_shared/review/crossai.md`.

### STEP 8 — close
Read `_shared/review/close.md`. Follow exactly.

### Profile shortcut branches
For infra-smoke / delta / regression / schema-verify / link-check profiles,
read `_shared/review/profile-shortcuts.md` and follow the branch matching
your profile. Do NOT execute full pipeline for shortcut profiles.

For `lens-dispatch.md` (the user's main concern):

```markdown
# Lens dispatch — STEP 4

## Why lens dispatch is mandatory

18 production lens probes cover security/UI/business surfaces. Without
forced dispatch, prior dogfood showed AI cherry-picks 3-4 lens, missing
critical findings (e.g., phase 3.2 filter pending bug at billing/topup-queue
went undetected for 2 rounds).

<HARD-GATE>
You MUST execute lens dispatch unless --skip-discovery flag is set AND
override-debt entry exists. Stop hook checks review.lens_phase.entered event;
missing = run-complete blocked.

You MUST NOT cherry-pick individual lens. LENS_MAP in spawn_recursive_probe.py
enforces the full element-class → lens mapping. Skipping a lens is impossible
unless eligibility gate (6 preconditions) declines it (audit trail in
.recursive-probe-skipped.yaml).
</HARD-GATE>

## Pre-dispatch sequence

1. Bash: `vg-orchestrator step-active phase2_5_recursive_lens_probe`
2. Bash: check lens-plan staleness (§5.1 item 10) — block if stale
3. Bash: run 6-precondition eligibility gate
4. AskUserQuestion (Claude) / inline (Codex): 3-axis preflight (RECURSION_MODE / PROBE_MODE / TARGET_ENV)
5. Bash: anti-forge guard verifies preflight wasn't skipped
6. Bash: emit `review.lens_phase.entered`

## Dispatch

Bash: `python scripts/spawn_recursive_probe.py --mode <PROBE_MODE> --recursion <RECURSION_MODE> --env <TARGET_ENV>`

This script (1,303 LOC, parallel-safe via per-tool subdir isolation):
- Reads REVIEW-LENS-PLAN.json
- Maps element-class → lens via LENS_MAP
- Spawns Gemini Flash workers per (element × lens × role) tuple
- Each worker writes `runs/<tool>/<element>-<lens>-<role>.json`
- Emits per-lens telemetry: `review.lens.<name>.dispatched` and `.completed`

## Aggregation

Bash: `python scripts/aggregate_recursive_goals.py`

Single-writer dedup, output `TEST-GOALS-DISCOVERED.md` with G-RECURSE-* entries.

## Step-end

1. Bash: emit `review.lens_phase.completed --lens_count_dispatched N --lens_count_completed M`
2. Bash: `vg-orchestrator mark-step review phase2_5_recursive_lens_probe`
```

## 6. Error handling, migration, testing, exit criteria

### 6.1 Error handling

All blocks follow blueprint pilot §4.5 (5-layer diagnostic). Review-specific:

- **Lens-plan staleness** → block with: "REVIEW-LENS-PLAN.json older than API-CONTRACTS.md. Run `python scripts/review-lens-plan.py` to regenerate, then retry."
- **Lens phase skipped without override** → Stop hook block: "review.lens_phase.entered event missing. Either invoke step phase2_5_recursive_lens_probe or add --skip-discovery flag with override-debt entry."
- **Per-lens dispatch failure** → recorded as `review.lens.<name>.dispatched` event with status=failed; aggregator counts; if failure rate >50% → block with retry suggestion.

### 6.2 Migration

- Existing 30 PrintwayV3 review runs: stand as-is.
- Existing tests: pass.

### 6.4 Exit criteria — review pilot PASS requires ALL of:

1. Tasklist visible in Claude Code UI immediately after invocation
2. `review.native_tasklist_projected` event count ≥ 1
3. `review.lens_phase.entered` + `review.lens_phase.completed` events both present (or `--skip-discovery` flag with override-debt entry)
4. Per-lens dispatch events: count ≥ count of lens in REVIEW-LENS-PLAN.json (no cherry-pick)
5. RUNTIME-MAP.json + GOAL-COVERAGE-MATRIX.md + REVIEW-LENS-PLAN.json + scan-*.json all written with content_min_bytes
6. Browser-discoverer subagent invocation event present (Task tool fired for phase2)
7. Phase4 verdict ref loaded (verify Read tool fired on `verdict/<branch>.md` matching profile)
8. **Phase 3.2 dogfood criterion**: lens dispatch detects the filter pending bug at billing/topup-queue (must surface in TEST-GOALS-DISCOVERED.md or findings)
9. CrossAI review runs (existing behavior, defer refactor — accept whatever produces; not a fail criterion this round)
10. Stop hook fires without exit 2
11. Manual: simulate skip lens phase → Stop hook blocks with diagnostic stderr
12. Stop hook unpaired-block-fails-closed test passes

Review pilot FAILS if any criterion missed. Most critical: criterion 8 (dogfood detection). Failure path: redesign lens dispatch enforcement.
