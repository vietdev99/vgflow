# `/vg:review` Ergonomics — Systematic Fix Design (3 bugs)

**Date:** 2026-05-04
**Source:** PV3 dogfood discovery (review run `4fe39741`, phase 4.1)
**Status:** Approved by sếp Dũng (5 design choices confirmed 2026-05-04)

---

## Goal

Fix 3 systemic UX failures in `/vg:review` discovered during PrintwayV3 (PV3) dogfood:

1. **Bug A** — Review halts on detection without giving the user any choice; user must restart from scratch every iteration.
2. **Bug B** — Native tasklist contract is generated by the orchestrator but the AI agent never projects it via `TodoWrite`; user sees only a plain markdown table.
3. **Bug C** — Findings get ad-hoc IDs (e.g. AI invented "E-001" for a missing endpoint); user has no way to interpret prefixes.

Each bug is independent. Fix each one in its own task; share infrastructure where it makes the system simpler, not where it forces coupling.

## Non-goals

- Refactor blocking-gate pattern outside `commands/vg/review.md` in this spec. Build / test / accept / scope use similar patterns and will inherit the wrapper, but their refactor is a separate follow-up plan once the wrapper proves itself in review.
- Migrate historical `REVIEW-FEEDBACK.md` files to the new finding-ID namespace. Forward-only — historical files stay as-is.
- Fix the underlying P3.5 carry-over bug in PV3 (missing `billing-summary` handler). That's the validation target for the new STEP 5.5 auto-fix-loop, not part of this design.

## Bug A — Review hard-block without `AskUserQuestion`

### Current behavior

Review's preflight + Phase 2a.5 (API contract probe) + Phase 4 (drift / matrix / replay) detect issues, then:

```bash
echo "⛔ <gate> BLOCK — fix path: <hint>"
emit-event review.<X>_blocked
exit 1
```

Locations (grep `exit 1` in `commands/vg/review.md`): preflight_invariants_blocked, api_precheck_blocked, asserted_drift_blocked, matrix_staleness_blocked, replay_evidence_blocked, env_mode_gate, foundation_drift, plus 13+ more — ~20 paths total.

The user must read the fix hint, manually fix, then `/vg:review --retry-failed` from the start. PV3 events.db shows 5 review re-runs in one debug session before the user even understood E-001 was a missing endpoint.

### Design decision (confirmed)

Introduce `commands/vg/_shared/lib/blocking-gate-prompt.sh` — a single shell helper that any blocking gate calls instead of hard-exiting. The helper takes `(gate_id, fix_hint, severity, evidence_path)` and presents a 4-option `AskUserQuestion` to the controller AI:

| Option | Effect |
|---|---|
| `[a] Auto-fix now` | Spawn `general-purpose` subagent with delegation contract similar to STEP 5.5 in-scope-fix-loop. Max 3 attempts. Subagent edits code (or runs migrations / curls / writes handlers) until the gate's validator passes. On PASS → re-run the gate; on FAIL → re-prompt this same 4-option question with subagent's repair packet attached. |
| `[s] Skip with override` | Emit `override.used --flag <skip-flag>`, log to override-debt register, mark step done, continue review. Requires `--override-reason=<text>` from the user. |
| `[r] Route to /vg:amend` | When the gate's root cause is a CONTEXT decision (e.g. drift register entry), exit review with a clean handoff message: "this needs `/vg:amend <phase>` first". Telemetry: `review.routed_to_amend`. |
| `[x] Abort review` | Clean exit. Run-complete event emits with `outcome: aborted_by_user`. Stop hook accepts (no unhandled blocks because user explicitly chose abort). |

### Pilot scope

Refactor only `commands/vg/review.md` to call the wrapper at every `exit 1` site. Build / test / accept refactor in follow-up tasks once the wrapper API proves stable.

### Subagent contract for option `[a]`

Same `Agent(subagent_type="general-purpose", prompt=...)` pattern as STEP 5.5, with adjustments for review context:

- Subagent runs against the **deployed app** (not source code) when the gate is API-related (`api_precheck`, `asserted_drift`, `replay_evidence`). It can curl, write handlers, restart the API service.
- Subagent runs against **artifacts** when the gate is drift / matrix-staleness — edits CONTEXT.md / drift register / RUNTIME-MAP.json directly.
- Max 3 attempts. After attempt 3 → return `{"status": "UNRESOLVED", "iterations": 3, "repair_packet": {...}}`. Wrapper presents the repair packet to the user via the same 4-option question.
- Honors `phase_ownership` allowlist from Task 8 — same scope guard as build-fix-loop.

## Bug B — Native tasklist not projected

### Current behavior

`commands/vg/review.md` STEP 0c writes `${PHASE_DIR}/.vg/runs/<run_id>/tasklist-contract.json` (schema=`native-tasklist.v2`, populated checklists). The AI agent is supposed to read it and call `TodoWrite` to project the checklist into the chat UI. Instead, the AI ignores the contract and renders a plain markdown table directly in its response.

PV3 events.db shows 15 historical `PreToolUse-tasklist` block.handled events — meaning the hook does fire when the AI tries to bypass, but the AI handles the block then continues without projecting the tasklist anyway.

### Design decision (confirmed)

Hard-gate via PreToolUse hook, not soft instruction.

Pattern adapted from `commands/vg/_shared/scope/preflight.md` line 591-617: orchestrator emits `tasklist-contract.json`, then a PreToolUse-Bash hook chặn `step-active` calls until `${PHASE_DIR}/.vg/runs/<run_id>/.tasklist-projected.evidence.json` exists. The evidence file is created by the orchestrator's `tasklist-projected --adapter claude` subcommand AFTER the AI has emitted the projected event (i.e. proven `TodoWrite` was called).

Specifically:

1. Add to `scripts/hooks/vg-pre-tool-use-bash.sh`: when the bash command contains `step-active` AND the run is `vg:review` AND `.tasklist-projected.evidence.json` is missing AND the corresponding `tasklist-contract.json` exists → BLOCK with cause "review must call TodoWrite to project tasklist before any step-active".
2. Add to `commands/vg/review.md` slim entry: between `run-start` and the first `step-active`, add a mandatory block: "Read `${PHASE_DIR}/.vg/runs/${RUN_ID}/tasklist-contract.json`. Call `TodoWrite` with the checklists array. Then run `vg-orchestrator tasklist-projected --adapter claude`."
3. New telemetry: `review.tasklist_projection_skipped` (emitted by hook on BLOCK) — gives us a metric to detect AI bypass attempts.

The scope command already has this pattern working (events.db confirms `scope.native_tasklist_projected` fires correctly). Extending to review is a copy-and-adapt of that working code.

## Bug C — Finding ID namespace

### Current behavior

When AI scanners (Phase 2a.5 API probe, Phase 4 drift checks, etc.) emit a finding, they invent ad-hoc IDs: "E-001", "Issue #3", "Finding A". The user sees "E-001" in the verdict and asks "what is E-001?" — the AI itself doesn't know what the prefix means because it's not standardized.

### Design decision (confirmed)

Standardize finding-ID prefix per category in `scripts/lib/scanner_report_contract.py` (already exists for the scanner-report-contract). The set:

| Prefix | Category | Source |
|---|---|---|
| `EP-` | Endpoint | API contract probe (Phase 2a.5), call-graph (Task 3 L4a-i) |
| `D-` | Drift | Asserted-rule drift, foundation drift, matrix staleness |
| `R-` | Rule | Bootstrap-rule violation, scope-matched rule (Task 11) |
| `G-` | Goal | Goal-comparison miss (Phase 4) |
| `F-` | Foundation | FOUNDATION-DRIFT entries |
| `S-` | Schema | Contract-shape mismatch (Task 4 L4a-ii) |
| `T-` | Telemetry | Missing required event |

ID format: `<prefix>-<3-digit-zero-padded>` (e.g. `EP-001`, not `E-001` or `EP-1`).

### Validator

`scripts/validators/verify-finding-id-namespace.py` — runs at end of Phase 4. Greps `REVIEW-FEEDBACK.md` for finding headers, validates each ID matches `^(EP|D|R|G|F|S|T)-\d{3}$`. Non-conforming → emit `review.finding_id_invalid` (warn-tier, not BLOCK in this rollout — gradual adoption).

Phase 2: after 30 days, promote to BLOCK once historical `REVIEW-FEEDBACK.md` are no longer being read.

### Backward compat

Forward-only. Historical files stay as-is (E-001 etc remain in old REVIEW-FEEDBACK.md). The validator runs only on the current phase's REVIEW-FEEDBACK.md, not on archived ones.

## Architecture summary

```
review.md slim entry
  ├── run-start
  ├── [NEW Bug B] read tasklist-contract.json + TodoWrite + tasklist-projected
  │      ↑ enforced by PreToolUse hook
  ├── step-active 0a / 0b / 0c (preflight)
  │      └── on BLOCK detected → call blocking-gate-prompt.sh
  │            ├── option [a] auto-fix subagent (max 3 attempts)
  │            ├── option [s] skip with override → continue
  │            ├── option [r] route to /vg:amend → exit clean
  │            └── option [x] abort → run-complete with aborted_by_user
  ├── ... 19+ other gates same pattern ...
  ├── Phase 4 goal_comparison
  │      └── new validator: verify-finding-id-namespace.py
  └── close
```

## Component boundaries

- `scripts/lib/blocking-gate-prompt.sh` — pure helper, no review.md state. Bash function, callable as `blocking_gate_prompt <gate_id> <fix_hint_file> <severity> [evidence_path]`. Returns user choice as exit code (0=fixed, 1=skip, 2=amend, 3=abort).
- `commands/vg/_shared/build/in-scope-fix-loop-delegation.md` — already exists from Task 10. Bug A reuses this contract; review wrapper passes it as `prompt` to spawned subagent.
- `scripts/validators/verify-finding-id-namespace.py` — single Python validator, reusable by review/test/accept on REVIEW-FEEDBACK.md / TEST-REPORT.md / UAT.md.
- `scripts/hooks/vg-pre-tool-use-bash.sh` — extend existing hook, don't add new hook. Add a single conditional block; fail-open if review run not active.

## Data flow

### Bug A example (api_precheck_blocked)

1. Phase 2a.5 detects 1/27 missing endpoint, writes evidence to `${PHASE_DIR}/.vg/api-precheck-evidence.json`.
2. Old: `emit api_precheck_blocked + exit 1`.
3. New: source `blocking-gate-prompt.sh`, call `blocking_gate_prompt "api_precheck" "${PHASE_DIR}/.vg/api-precheck-evidence.json" error`.
4. Wrapper writes a structured AskUserQuestion JSON to stdout with the 4 options.
5. AI controller reads, presents to user, gets answer.
6. Wrapper exits with the corresponding code.
7. Calling site branches: 0 → re-run gate validator, 1 → emit override + continue, 2 → exit with route-to-amend message, 3 → exit with abort.

### Bug B example

1. `run-start` writes `tasklist-contract.json` (already happens).
2. AI hits `Bash` tool with `vg-orchestrator step-active 0a_env_mode_gate`.
3. PreToolUse-bash hook checks `.tasklist-projected.evidence.json` — missing → BLOCK with cause "TodoWrite required first".
4. AI reads block, calls `TodoWrite(...)` with the contract's checklists array.
5. AI calls `vg-orchestrator tasklist-projected --adapter claude` — orchestrator validates the recent TodoWrite payload matches contract checksum, writes `.tasklist-projected.evidence.json`.
6. AI retries the original `step-active` — PreToolUse passes.

## Error handling

- Wrapper subagent failure (option `[a]`): subagent returns `{"status": "UNRESOLVED"}` → wrapper re-prompts user with the repair packet attached. User picks again from same 4 options. No infinite loop because `[a]` only spawns once per prompt instance; if user picks `[a]` again it's a fresh subagent invocation.
- TodoWrite hook bypass attempt: hook BLOCKs with diagnostic, AI must call TodoWrite. If AI bypasses 3+ times in same run → escalate via existing block-correlator (Task 32) — manual intervention.
- Finding-ID validator: warn-tier means non-conforming → emit event, don't fail review. Telemetry gives us adoption curve.

## Testing plan

Each bug gets a self-contained test under `tests/`:

- `tests/test_blocking_gate_prompt.py` — fixtures simulate a calling site emitting evidence, mock user answer via env var `VG_TEST_USER_ANSWER`, assert wrapper exits with right code + emits right events.
- `tests/test_review_tasklist_projection_hook.py` — synthetic `.vg/active-runs/<id>.json` + `tasklist-contract.json`, run hook with `step-active` command, assert BLOCK before evidence file, PASS after.
- `tests/test_finding_id_namespace_validator.py` — synthetic REVIEW-FEEDBACK.md with mixed IDs, assert validator catches non-conforming + accepts conforming.

Plus integration test `tests/test_review_p4_dogfood_replay.py`: synthetic phase with 1 missing endpoint, run review, assert wrapper triggers and 4 options present.
