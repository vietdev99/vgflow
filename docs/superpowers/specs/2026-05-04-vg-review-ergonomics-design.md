## Goal

Fix 7 systemic UX + design failures spanning `/vg:review` and the blueprint→build envelope, all surfaced during PrintwayV3 (PV3) dogfood:

**Review-side (Tasks 33-36, "Track A"):**

1. **Bug A** — Review halts on detection without giving the user any choice; user must restart from scratch every iteration.
2. **Bug B** — Native tasklist contract is generated by the orchestrator but the AI agent never projects it via `TodoWrite`; user sees only a plain markdown table.
3. **Bug C** — Findings get ad-hoc IDs (e.g. AI invented "E-001" for a missing endpoint); user has no way to interpret prefixes. Codex round 1 caught 4 prefix collisions — see Bug C section for the corrected namespace.
4. **Bug D** — Review defers UI lens probes to `/vg:test` instead of running them; user reads "DEFERRED" rows instead of seeing actual UI bug findings. Task 26 (lens dispatch enforcement) shipped the infrastructure; Bug D wires it into review.md Phase 2.5.

**Build/blueprint-side (Tasks 37-40, "Track B"):**

5. **Bug E** — Blueprint produces UI-MAP / CRUD-SURFACES / LENS-WALK / RCRURD invariants but the build subagent envelope only carries some of them. Build FE/mutation tasks guess kit pattern + invariants instead of reading them.
6. **Bug F** — API-CONTRACTS.md is BE-only (4 blocks: auth, schemas, errors, sample). FE codegen has nothing to read for canonical URL, UI states, query param schema, cache invalidation, optimistic semantics, toast text. Result: "wrong URLs even though code đã gen rồi." Add BLOCK 5 = FE consumer contract.
7. **Bug G** — RCRURD invariant schema (Task 22) only handles ONE write+read cycle; the actual lens-form-lifecycle pattern is **R-C-R-U-R-D-R** (Read empty → Create → Read populated → Update → Read updated → Delete → Read after delete) = 7 ops, 4 reads. Extend schema with `lifecycle_phases[]` array.
8. **Bug H** — No artifact captures multi-actor cross-role workflows (e.g. User creates request → Admin approves → User sees result). RCRURDR is single-actor; cred switches + multi-view sequences need a Workflow artifact. Add `WORKFLOW-SPECS.md` blueprint output + per-flow split + Pass 3 subagent.

**Build coordination mitigations (Tasks 41-43, "Track C"):**

9. **Bug I (M1)** — `pre-executor-check.py` capsule has no `actor_role` / `workflow_id` / `write_phase` fields. Build subagent doesn't know which side of a multi-actor workflow it's implementing. Extend capsule fields.
10. **Bug J (M2)** — `wave-N-context.md` cites cross-task constraints by file/field only ("Task 7 + Task 8 reference these enum values"), NOT by workflow/state. Cross-wave workflow coordination invisible — wave 5 USER subagent doesn't know wave 8 ADMIN subagent is downstream. Add cross-wave workflow references.
11. **Bug K (M3)** — `waves-delegation.md` prompt template has no `<workflow_context>` block. Per-slice size validator is 30 KB warn (advisory, not BLOCK). Add prompt block + promote validator to per-unit ≤ 5K-token BLOCK.

Each bug is independent at the task level. Tracks A, B, C can run partly in parallel (see sequencing graph).

## Glossary (terms locked)

To avoid the 3-parallel-name confusion Codex finding #46 flagged:

- **RCRURDR** — canonical name for the 7-op lifecycle (Read empty → Create → Read populated → Update → Read updated → Delete → Read after delete). Used in spec, schema field `lifecycle: "rcrurdr"`, lens-form-lifecycle prompt, scanner-report-contract `goal_class`.
- **rcrurd** — the legacy single write+read invariant (Task 22 default). Schema field `lifecycle: "rcrurd"` (= just 1 phase: write + read + assert). Backward-compat default.
- **`crud-roundtrip`** — the transition kit name (existing, in `_shared/transition-kits/`). Treated as alias for RCRURDR. The kit's prompt instructs operators to capture all 7 lifecycle ops; the `goal_class: crud-roundtrip` in scanner-report-contract maps 1:1 to lifecycle: rcrurdr.
- **`lifecycle_phases`** — schema field name in `rcrurd_invariant.py` (extended in Task 39, NOT renamed). When `lifecycle: rcrurdr`, this list MUST contain exactly 7 phase entries (read_empty, create, read_populated, update, read_updated, delete, read_after_delete). When `lifecycle: partial` (Codex round-3 v3.2 lock), it MUST contain exactly the goal-type-specific subset (3 phases for create_only/update_only/delete_only).
- **`lifecycle: partial`** — discriminator value (Codex round-3 v3.2 lock) for sub-lifecycle goals (`goal_type ∈ {create_only, update_only, delete_only}`). Same per-phase iterator as rcrurdr but smaller phase list.
- **`write_phase`** — capsule field (Bug I, Task 41). Indicates which WRITE op the task implements: `create | update | delete | null`. NOT the same as `lifecycle_phases` (which has 7 RCRURDR ops). Read-only tasks have `write_phase: null`. Subagent uses this to pick the right RCRURDR phase block from per-goal invariant.

## Bug A — Review hard-block without `AskUserQuestion`

### Current behavior

Review's preflight + Phase 2a.5 (API contract probe) + Phase 4 (drift / matrix / replay) detect issues, then:

```bash
echo "⛔ <gate> BLOCK — fix path: <hint>"
emit-event review.<X>_blocked
exit 1
```

Locations (grep `exit 1` in `commands/vg/review.md`): preflight_invariants_blocked, api_precheck_blocked, asserted_drift_blocked, matrix_staleness_blocked, replay_evidence_blocked, env_mode_gate, foundation_drift, plus 13+ more — ~20 paths total.

The user must read the fix hint, manually fix, then `/vg:review --retry-failed` from the start. PV3 events.db shows 5 review re-runs in one debug session before the user even understood E-001 was a missing endpoint.

### Design decision (Codex fix B2: 2-leg contract)

A bash script cannot invoke `AskUserQuestion` directly (that is a tool call from the AI controller, not a shell call). Wrapper is therefore split into **2 legs** the AI controller orchestrates:

**Leg 1 — `blocking_gate_prompt_emit`** (bash, called by the gate's calling site):
- Inputs: `<gate_id> <fix_hint_file> <severity> [evidence_path]`
- Output: structured JSON on stdout describing the 4 options (option keys + display text + subagent contract paths)
- Side effects: writes `.vg/runs/<run_id>/.gate-prompt-${gate_id}.json` with full prompt context for resume
- Exit code: 0 if prompt emitted successfully, 64 if input invalid (BSD sysexits convention reserves 64+ for wrapper internal errors)

**AI controller step** (between Leg 1 and Leg 2):
- Reads stdout JSON
- Invokes `AskUserQuestion` with the title + options
- Captures user's choice letter

**Leg 2 — `blocking_gate_prompt_resolve`** (bash, called after user answers):
- Inputs: `<gate_id> --user-choice=<a|s|r|x> [--override-reason=<text>]`
- Side effects: based on choice, dispatch:
  - `a` → spawn subagent (Agent tool call, instructed by AI controller per option JSON), wait for return, validate, re-emit (Leg 1) on UNRESOLVED for re-prompt
  - `s` → emit `override.used` + log debt + emit `review.gate_skipped_with_override`
  - `r` → emit `review.routed_to_amend` + write handoff message
  - `x` → emit `review.aborted_by_user` + clean exit
- Exit codes: 0 = fixed (gate now passes, calling site continues), 1 = skip-with-override (continue), 2 = route-to-amend (caller exits cleanly), 3 = abort (caller exits), 4 = re-prompt-needed (subagent UNRESOLVED, AI controller must re-call Leg 1 with appended repair packet)

### Option matrix

| Option | Effect |
|---|---|
| `[a] Auto-fix now` | AI controller spawns `general-purpose` subagent per the **NEW** `commands/vg/_shared/lib/review-fix-loop-delegation.md` contract (NOT the build-shaped Task 10 contract — review fixes against deployed app + artifacts, not source-file ownership window). Max 3 attempts. On PASS → re-run gate validator → if still fails, exit 4 (re-prompt with new repair packet). |
| `[s] Skip with override` | Emit `override.used --flag <skip-flag> --severity <wrapper-sev-mapped-to-debt-vocab>`, log to override-debt register, mark step done, continue. Requires `--override-reason=<text>` (≥50 chars). |
| `[r] Route to /vg:amend` | When the gate's root cause is a CONTEXT decision (e.g. drift register entry), exit review with handoff message: "this needs `/vg:amend <phase>` first". Telemetry: `review.routed_to_amend`. |
| `[x] Abort review` | Clean exit. Run-complete event emits with `outcome: aborted_by_user`. Stop hook accepts (no unhandled blocks because user explicitly chose abort). |

### Severity vocabulary mapping (Codex finding #76)

Wrapper input vocab is `error|warn|critical` (review.md gates). Override-debt vocab is `critical|high|medium`. Mapping:
- wrapper `critical` → debt `critical`
- wrapper `error` → debt `high`
- wrapper `warn` → debt `medium`

Locked in `commands/vg/_shared/lib/blocking-gate-prompt-contract.md`.

### `--non-interactive` mode (Codex finding #81)

When `$ARGUMENTS` contains `--non-interactive` (CI runs), Leg 1 short-circuits: skip AskUserQuestion entirely, behave as if user picked `[x] Abort review`. Emit `review.aborted_non_interactive_block` (warn-tier). Rationale: silent auto-fix in CI is dangerous; skip-with-override would silently accumulate debt. Loud abort is correct CI semantics.

### Pilot scope

Refactor `commands/vg/review.md` only — at every `*_blocked` event-emitting `exit 1` site (≈ 13 paths matching the frontmatter `must_emit_telemetry` warn-tier `review.<X>_blocked` declarations). Non-blocking `exit 1` paths (precondition guards, missing-file aborts) stay as-is. Build/test/accept generalization in follow-up plan.

### Selection rule for refactor sites (Codex finding #1)

A `exit 1` site is in scope IFF all three:
1. Immediately preceded by `emit-event "review.<X>_blocked"` call (or within 5 lines)
2. The `<X>` matches one of the 13 declared telemetry events: `api_precheck_blocked`, `preflight_invariants_blocked`, `matrix_staleness_blocked`, `matrix_evidence_link_blocked`, `evidence_provenance_blocked`, `asserted_drift_blocked`, `mutation_submit_blocked`, `rcrurd_preflight_blocked`, `rcrurd_depth_blocked`, `rcrurd_post_state_blocked`, `edge_case_variant_blocked`, `tasklist_projection_skipped` (Task 34's), `routed_to_amend` (Task 33's emit point — included for symmetry, not refactored)
3. Site has a user-facing fix-path message (`echo "⛔ ... fix:"` or similar)

Sites NOT in scope: missing-file guards, schema-load failures, internal assertion failures (these stay `exit 1` — they are bugs, not gate decisions).

### Subagent contract for option `[a]` (Codex fix B5)

NEW file: `commands/vg/_shared/lib/review-fix-loop-delegation.md`. Different from Task 10's build-fix-loop contract:

- Subagent runs against the **deployed app** when the gate category is API-related (`api_precheck`, `asserted_drift`, `replay_evidence`, `mutation_submit`). It can curl handlers, write FE/BE source code, restart API service.
- Subagent runs against **artifacts** when the gate is drift/matrix-related (`matrix_staleness`, `foundation_drift`, `rcrurd_post_state`) — edits CONTEXT.md, drift register, RUNTIME-MAP.json.
- Max 3 attempts. After attempt 3 → return `{"status": "UNRESOLVED", "iterations": 3, "repair_packet": {...}}`. Wrapper Leg 2 exits with code 4 (re-prompt).
- Phase-ownership allowlist: review subagent has WIDER scope than build subagent (deployed-app + artifacts) but still constrained to the current phase + its dependencies. Excludes other phases' artifacts.
- Forbidden: editing API-CONTRACTS.md → return UNRESOLVED with `blocked_by="contract_amendment_required"` → wrapper short-circuits to option `[r]` (auto-route to amend, no re-prompt).

## Bug B — Native tasklist not projected

### Current behavior

`commands/vg/review.md` writes `.vg/runs/<run_id>/tasklist-contract.json` (schema=`native-tasklist.v2`, populated checklists). The hook in `scripts/hooks/vg-pre-tool-use-bash.sh` ALREADY enforces evidence file existence before allowing any `step-active`. The hook's `Required fix` text already mentions `TodoWrite`. So the hook is not the bug — its diagnostic just isn't strong enough to interrupt AI's habit pattern.

PV3 events.db shows 15 historical `PreToolUse-tasklist` block.handled events. AI hits block, emits `vg.block.handled` (loop closes), then continues forward — but evidence file still missing, next `step-active` blocks again. AI ends up looping or abandoning step-active entirely.

### Design decision (Codex fix: hook diagnostic upgrade + slim-entry positioning + telemetry only)

The hook does NOT need new conditional logic — it already gates by evidence file. What's needed is:

1. **Slim-entry positioning** — review.md slim entry must reference `_shared/lib/tasklist-projection-instruction.md` BEFORE any `<step name=...>` block (currently the TASKLIST_POLICY section is at lines 268-327 but separated from the imperative). Test asserts byte position: instruction marker MUST appear before the first `<step ` tag (Codex finding #14 — first occurrence of "step-active" is too loose; tighten to `<step ` opening tag).
2. **Hook diagnostic upgrade** — replace existing `Required fix` text in `emit_block()` with a stronger, more imperative + more specific block: name the exact tool (`TodoWrite`), the exact subcommand (`vg-orchestrator tasklist-projected --adapter claude`), and the exact file path (the evidence file location). Test asserts the upgraded diagnostic contains the new exact phrase that did NOT appear in the old diagnostic — so red-then-green TDD works (Codex finding #11).
3. **New telemetry event** — `<command>.tasklist_projection_skipped` (warn-tier) emitted by hook on BLOCK. AI bypass attempts become observable in `/vg:gate-stats`.

### Path canonicalization (Codex finding #51)

Evidence file path = **`.vg/runs/<run_id>/.tasklist-projected.evidence.json`** (repo-relative, NOT `${PHASE_DIR}/.vg/runs/...`). The existing hook (`vg-pre-tool-use-bash.sh:23`) is repo-relative and that's correct. Spec was wrong; fixed throughout this revision.

### Multi-command effect

The hook fires for ALL commands (build/test/scope/accept use the same evidence-file check). When `command_from_run="vg:review"`, hook emits `review.tasklist_projection_skipped`. When `command_from_run="vg:build"`, hook emits `build.tasklist_projection_skipped`, etc. Single hook + per-command event = all VG commands gain projection telemetry simultaneously. (Codex finding #16: this is the right scope.)

### Test that AI bypass remains expensive (Codex finding #19)

Add test: simulate the bypass — AI emits `vg.block.handled` without creating evidence file. Re-call hook → must STILL block (because evidence file still missing). Asserts that emitting handled does NOT bypass the gate, only TodoWrite + orchestrator subcommand can.

## Bug C — Finding ID namespace (Codex fix B1: re-namespace to avoid collisions)

### Current behavior

When AI scanners emit a finding, they invent ad-hoc IDs: "E-001", "Issue #3", "Finding A". The user sees "E-001" in the verdict and doesn't know what category it is.

### Design decision (REVISED after Codex finding #20-24)

Original prefix set (`EP|D|R|G|F|S|T`) had **5 collisions** with existing IDs in the codebase:
- `F-001` already used in `scripts/replay-finding.py` for replay finding IDs
- `D-XX` already used as decision IDs (`P{phase}.D-XX`) in CONTEXT.md
- `G-XX` already used as goal IDs (`G-04`, `G-141`, `G-W10-05`) — and existing goals are 2-digit, the original `\d{3}` regex would have rejected them
- `R-XX` overlaps with `BR-` business rules
- `OD-XX` already used for override-debt entries

**Re-namespaced** to non-colliding prefixes:

| Prefix | Category | Source | Width |
|---|---|---|---|
| `EP-` | Endpoint | API contract probe (Phase 2a.5), call-graph (Task 3 L4a-i) | 3-digit |
| `DR-` | Drift | Asserted-rule drift, foundation drift, matrix staleness | 3-digit |
| `RV-` | Rule-violation | Bootstrap-rule violation, scope-matched rule (Task 11) | 3-digit |
| `GC-` | Goal-comparison | Goal-comparison miss (Phase 4) — distinct from goal authoring `G-XX` | 3-digit |
| `FN-` | Foundation | FOUNDATION-DRIFT entries — distinct from replay-finding `F-XXX` | 3-digit |
| `SC-` | Schema | Contract-shape mismatch (Task 4 L4a-ii) | 3-digit |
| `TM-` | Telemetry | Missing required event | 3-digit |

Validator regex: `^(EP|DR|RV|GC|FN|SC|TM)-\d{3}$` (all 2-letter prefixes, no 1-letter to keep visual distinction from existing `D-/G-/F-/R-` namespaces).

### Validator

`scripts/validators/verify-finding-id-namespace.py` — runs at end of Phase 4. Greps `REVIEW-FEEDBACK.md` for finding-section headers.

**Regex** (Codex round-2 Amendment E — match real PV3 format):

```regex
^###\s+(EP|DR|RV|GC|FN|SC|TM)-\d{3}\s+\[(?:CRITICAL|MAJOR|MINOR|INFO)\]\s
```

This matches `### EP-001 [MAJOR] GET /api/...` (real format used in current PV3 RUNTIME-MAP.md). The earlier draft regex `^### (...)-\d{3}: ` would silently report ZERO findings on real artifacts (severity is in brackets, no colon).

Test fixtures in `tests/test_finding_id_namespace.py` MUST use the real format.

When AI emits a non-conforming ID (e.g. legacy `E-001`), validator emits `review.finding_id_invalid` (warn-tier) AND prints a fix-suggestion: "did you mean `EP-001`?" (mapping single-letter → 2-letter — Codex finding #28).

### Warn-tier rollout — observable success metric (Codex finding #27)

Promote validator from warn → BLOCK when:
- Zero `review.finding_id_invalid` events for 14 consecutive runs across ≥2 projects
- AND no manual override/skip used during that window

Tracked in `/vg:gate-stats finding-id-namespace --since 14d`. Promotion is operator-triggered (edit validator severity in code), not auto-time-based.

### Backward compat

Forward-only. Historical files stay as-is. Validator runs only on the current phase's REVIEW-FEEDBACK.md, not on archived ones.

## Bug D — Review defers UI lens probes to /vg:test (wire Task 26 into review.md Phase 2.5)

### Current behavior

PV3 review run output:

```

Add `recommended_worker_tier`, `worker_complexity_score`, `min_actions_floor`, `min_evidence_steps`, `required_probe_kinds`, `fallback_on_inconclusive` to all 19 `commands/vg/_shared/lens-prompts/lens-*.md` files. - **Task 36b** — Wire Task 26 into review.md Phase 2.5. Coverage gate failure routes through Task 33's wrapper (4-option AskUserQuestion). LENS-COVERAGE-MATRIX.md rendered for user via Task 26's existing aggregator.

### Skip-mode interaction (Codex finding #32)

Existing `--probe-mode skip` writes `.recursive-probe-skipped.yaml`. Coverage gate guards: if `.recursive-probe-skipped.yaml` exists, gate skipped (legitimate user decision). This preserves existing escape hatch.

### BLOCK on MISSING vs wrapper-routing (Codex finding #31)

`verify-lens-runs-coverage.py` returns exit 1 when MISSING dispatches detected. Task 36b wires this into review.md as a Task 33 wrapper call site (gate_id = `lens_coverage_blocked`). User gets 4-option AskUserQuestion when MISSING:

- `[s] Skip with override` → emit override-debt, continue (lens findings missed)
- `[r] Route to /vg:amend` → not applicable (no decision change), wrapper short-circuits to `[s]`
- `[x] Abort` → standard

## Bug E — Build envelope incomplete (Codex finding #33: scope shrunk)

### Current state (Codex finding #33 corrected the conversation context)

`commands/vg/_shared/build/waves-delegation.md:30-31, 61-62` ALREADY pass `ui_map_path` + `view_components_path` in envelope and `@${PHASE_DIR}/UI-MAP.md (if exists)` in prompt template. So UI-MAP and VIEW-COMPONENTS are NOT the gap.

Actual missing artifacts (verified by grep):

1. **CRUD-SURFACES.md** — produced by blueprint, consumed only by `/vg:test`. Build mutation tasks don't see kit:crud-roundtrip pattern → handler shape drift.
2. **LENS-WALK seeds** — produced by blueprint Step `2b5e_a_lens_walk`, consumed only by `/vg:review`. Build can't pre-test against lens scenarios.
3. **RCRURD invariant per task** — produced by Task 22 schema (per-goal), but envelope has no field for per-task invariant slice. Build FE/mutation task subagent doesn't see invariants → forgets to insert `expectReadAfterWrite()` from Task 24 helper.

### Design decision (Codex finding #34-35: per-task slicing)

CRUD-SURFACES is per-phase but build envelope is per-task. Loading whole CRUD-SURFACES.md into every task's envelope inflates context budget. Solution: per-task slice via `vg-load`:
- `vg-load --phase N --artifact crud-surfaces --task NN` → returns only resources/operations the task touches
- `vg-load --phase N --artifact lens-walk --goal G-NN` → returns only lens variants for goals the task implements
- `vg-load --phase N --artifact rcrurd-invariant --task NN` → returns only invariants for goals the task touches

Orchestrator pre-resolves per-task slices BEFORE spawning the executor (Codex finding #35: subagent doesn't know goal→task mapping; orchestrator does). Slice paths added to envelope:

```json
"crud_surfaces_slice_path": ".vg/runs/.../task-NN.crud-surfaces.md",
"lens_walk_slice_path": ".vg/runs/.../task-NN.lens-walk.md",
"rcrurd_invariants_paths": [".vg/runs/.../task-NN.rcrurd.G-04.yaml", ".vg/runs/.../task-NN.rcrurd.G-12.yaml"]
```

### Backward compat (Codex finding #36)

For in-flight phases with stale capsules: `vg-build-task-executor` SKILL.md must tolerate missing fields with graceful degradation — log warning, continue with reduced context. Old phases keep working; new phases get richer envelope.

## Bug G — RCRURDR full lifecycle (Codex fix: extend in place, not rename)

### Current state
